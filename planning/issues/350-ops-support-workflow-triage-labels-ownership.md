## Context
Once live, you need a consistent process for inbound issues and app store reviews.

## Goal
Stand up a lightweight support/triage workflow that scales with a small team.

## Expected behavior
- Users have a clear way to contact support
- Team has a consistent triage process
- Issues are routed/owned

## Acceptance criteria (Definition of Done)
- [ ] Create `docs/ops/support.md` with triage flow (severity, response times, escalation)
- [ ] Define labels for support requests and bugs (if missing)
- [ ] Define ownership rotation (weekly) and backlog hygiene rules
- [ ] Add a GitHub issue template for support requests (optional)
- [ ] Unit/widget/integration tests added or updated
- [ ] Telemetry added/updated (event names + key properties)
- [ ] Offline-first behavior verified (where applicable)
- [ ] Accessibility basics (labels, contrast, tap targets)

## Out of scope
- Full customer support platform (Zendesk, etc.).

## Implementation notes
- Keep it simple: email + GitHub triage for early stage.
- Include app store review response guidelines.
